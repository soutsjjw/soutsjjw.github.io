最佳化和調整

程式碼膨脹（code bloat）通常指的是程式碼中不必要的冗長、低效或浪費資源的情況。程式碼膨脹會導致應用程序更容易出現錯誤、回歸錯誤、邏輯不一致等問題，並使調試和測試變得更加困難和壓力重重。

要對抗程式碼膨脹，最有效的方法是採用”不要重複自己”（Don’t Repeat Yourself，DRY）原則，這是開發者在開發過程中應該遵循的原則。DRY原則有助於避免陷入”WET”（Write Everything Twice、We Enjoy Typing、Waste Everyone’s Time）方式，即重複撰寫代碼。

在這個章節中，將討論如何改善當前代碼中的一些冗長部分，以使其更符合DRY原則，從而有助於後續的調試和測試工作。

模板改進

在CityEditComponent和CountryEditComponent的模板文件中存在的程式碼膨脹問題。具體來說，提到了使用大量mat-error元素，有時一個輸入甚至會有多達三個mat-error元素，這導致代碼重複和冗長。這種方法在小型表單中可能有優勢，但對於大型表單或擁有多個表單的應用程序來說，會導致大量難以維護的源代碼，特別是在多語言應用程序中。

建議解決程式碼膨脹的方法是使用輔助方法，在組件類中集中底層邏輯，以代替重複驗證邏輯。這樣可以提高代碼的可維護性和效率。文章暗示將在後續內容中嘗試將這些輔助方法添加到與表單相關的Angular組件中。

表單驗證捷徑

CountryEditComponent 中添加表單驗證的捷徑，因為不得不檢查兩個ISO 3166-1國家代碼的許多可能的錯誤狀態，這導致了產生大量的mat-error元素。

getErrors()的函數返回與給定控制項的活動錯誤對應的錯誤訊息陣列，或者如果沒有錯誤，則返回一個空陣列。這些錯誤訊息可以根據呼叫者設定的displayName參數進行客製作。使用了switch語句，其中包含所有目前使用的驗證器，以及一個預設情況，用於處理不支持的驗證器。最終目標是建立一個集中的”捷徑”方法，以減少HTML模板的冗余代碼，並提高程式碼的可維護性。

類別繼承

面向對象編程（OOP）中的兩個核心概念：多態性和繼承。多態性允許實體具有多個形式，而繼承則允許我們通過派生一個對象或類來擴展它，同時保留相似的實現。

我們專注於繼承，在TypeScript中，通過繼承創建的子類會繼承父類的所有屬性和行為，但不包括構造函數、析構函數、重載運算符和私有成員。這種繼承概念非常適合優化程式碼，特別是當我們想要在多個類中實現相同的方法時。接下來，我們將實現這種繼承以提高代碼的可重用性。

DTO 類別——我們真的應該使用它們嗎？

我們已經知道 Country 實體類別和 CountryDTO 類別有多相似，我們是否可以在 CountryDTO 類別中繼承 Country 實體類別，避免重複四個屬性；或者僅將 TotCities 屬性添加到 Country 實體實體中，不使用 CountryDTO 類別。是可以這樣做的，我們沒有這樣做是因為先前的解決方案都帶有一些相關的設計和安全缺陷。讓我們盡力解決它們，並了解為什麼在能夠時應該盡量避免使用它們。

關注點分離

通常實體類別不應該包含僅為滿足客戶端需求而存在的屬性。當需要這樣的屬性時，建議創建一個中介類別，將實體與通過 Web API 傳送到客戶端的輸出對象分開。在此情境中，將 TotCities 屬性添加到實體類別中會破壞這種分離，因為這個屬性並不存在於 Countries 資料表中，僅是為了向前端發送額外的資料。同時，TotCities 屬性與已經存在的 Cities 屬性之間沒有關聯，如果我們啟用 EF Core 的急切加載模式並填充 Cities 屬性，TotCities 屬性仍將設置為零（反之亦然），這可能導致誤導性行為和實現錯誤。

安全考慮

將實體類別和客戶端API輸出類別分開保持獨立是一個明智的選擇，尤其是出於安全考慮。在處理包含個人或登錄資料的使用者資料表等情況下，僅從資料庫中取出所有內容並直接傳送到客戶端可能是不明智的。在複雜的情境下，建議向客戶端提供有限的資料並以受控的方式進行。因此，在.NET中，建立並使用較輕、更安全的DTO類別，而不是主要的實體類別，是實現這一目標的最有效方法。

DTO 類別與匿名類型

使用 DTO 類別：

  1. 結構化和型別安全： DTO 類別是具有結構的型別，你可以明確定義其屬性和類型。這提供了結構化的資料傳輸，同時也提供了型別安全。

  2. 重複使用性： DTO 可以在不同的地方重複使用，因為它們是類別。這有助於避免程式碼重複和維護的困難。

  3. 版本控制： 如果你的 API 需要進行版本控制，DTO 可以更容易地進行版本管理。你可以根據不同版本的 API 使用不同版本的 DTO。

  4. 易讀性和維護性： DTO 提供了更好的代碼可讀性和維護性，因為你可以清楚地看到資料的結構。

使用匿名類型：

  1. 簡潔性： 匿名類型通常比較簡潔，特別是對於小型的資料集。如果你只需返回一些簡單的資料，使用匿名類型可能會更輕量。

  2. 即時性： 如果你只是在特定的 API 請求中需要使用某些資料，而這些資料在其他地方不需要，使用匿名類型可以減少不必要的類別定義。

  3. 簡單的轉換： 對於一次性的資料傳輸，使用匿名類型可能更簡單，不需要定義額外的類別。

總的來說，對於大多數情況，使用 DTO 類別是推薦的。它提供了更好的結構、型別安全和可讀性。然而，如果你僅需傳輸少量的簡單資料，而不需要重複使用，使用匿名類型也是合理的。最終的選擇應取決於你的具體需求和項目設計。

保護實體

在ASP.NET Web API中，如果不想使用DTO類別，也不喜歡匿名類型，有一個可行的替代方案，即保護實體以防止它們向EF Core發送不正確的指令或通過RESTful API發送過多的資料。可以通過使用特定的Data Annotation屬性來實現這一目標，例如：

  • [NotMapped]：防止EF Core為該屬性建立資料列
  • [JsonIgnore]：防止屬性被序列化或反序列化
  • [JsonPropertyName(“name”)]：允許我們在 JSON 類別序列化和反序列化時覆蓋屬性名稱，覆蓋屬性名稱以及在 Program.cs 檔案中由 JsonNamingPolicy 設定指定的任何命名策略。

在先前的章節中使用了[JsonPropertyName]屬性，現在我們來介紹其他兩個特性。

[NotMapped] 和 [JsonIgnore] 特性

我們的操作概述如下：

  • 在實體代碼中實現了 TotCities 屬性，使用 [NotMapped] 特性，防止 EF Core 建立對應的資料列。
  • 在此過程中，編寫了額外邏輯，將 TotCities 屬性鏈接到 Cities 屬性值，以避免提供誤導性的訊息。
  • 將 [JsonIgnore] 特性添加到 Cities 屬性，以防止將這些訊息發送到客戶端，即使值為 null 也是如此。

使用 [JsonIgnore] 防止伺服器發送 Cities 屬性可能看起來有點多餘，因為它目前是 null。但我們採取這項預防措施，因為我們直接使用實體，而不是 DTO 類別或匿名類型，我們想實施一種限制性的資料處理方式。在不需要該屬性時，使用 [JsonIgnore] 是明智的，以確保我們不會洩露比必要更多的信息。這被描述為默認的資料保護方法，有助於保持 Web API 的控制，防止過多的信息共享。當然，我們隨時可以在需要時刪除這個特性。

“Swiss Army knives” 的缺點

我們剛才討論的「安全實體」方法可能聽起來像是一個有效的 DTO 替代方案，因為它允許我們透過在實體類別內部集中邏輯而撰寫較少的程式碼，使其成為一個「萬能工具」。然而，不幸的是，這種方法實際上並不像看起來那麼好，因為它會使我們的代碼庫暴露於一些我們應該避免的缺點。

這種「安全實體」方法可能看似是一個有效的 DTO 替代方案，但實際上存在著一些需要避免的問題：

  1. 違反單一職責原則（SRP）： 直接在實體類別中配置 JSON 輸出破壞了單一職責原則，導致一個模組、類別或函數負責太多功能。

  2. 潛在的資訊洩漏： 實體可能包含不應該對使用者或客戶端應用程式顯示的敏感資料，隱藏這些屬性可能導致原始碼複雜化並最終產生難以理解的程式碼。

  3. 演進的實體類別： 實體類別可能會隨時間演進，新屬性的添加需要開發者的注意，遺漏 [JsonIgnore] 屬性可能導致資訊洩漏。

  4. 性能影響： 使用 Include 方法填充 TotCities 欄位可能導致過度擷取，從資料庫轉移大量不必要的資料，對性能產生顯著影響。

總體來說，這種方法容易導致開發者錯誤，增加資訊洩漏的風險，而唯一的優勢是減少（有用的）類別的數量。因此，值得嗎？答案是否定的。我们需要的不是少數多功能的工具，而是多個編寫得當且可讀性高的類別，以最佳且最有效的方式處理它們所需的任務。

瑞士軍刀(Swiss Army knives)，有時也被稱為廚房水槽(kitchen sink)，是指開發者給予過度複雜的類別介面的名稱，這些介面被明確設計來滿足所有可能的需求，但這種做法通常使事情變得過於複雜，而不是簡化。這被視為不良實踐。

最後的想法

在這一節中，討論了 GetCountries() 方法的三種替代實現方式，分別是 CountryDTO、匿名類型和 Country。這些實現方式都可以在 GitHub 上的 /Controllers/CountriesController.cs 文件中找到。其中，我們將在後面的範例中使用 CountryDTO 方法，而其他兩種方法被註釋並僅供參考。建議根據其編程風格隨時切換這些方法，但同時強烈建議考慮使用 DTO 類別，以做出更負責任的選擇。

資料服務

到目前為止，我們已經建立了兩個網頁應用程式——HealthCheck和WorldCities，它們使用Angular框架實現了前端到後端的通訊。通訊是通過HttpClient類別實現的，這是Angular的內置HTTP API客戶端。HttpClient類別具有多個優點，包括測試功能、請求和響應類型物件、攔截請求和響應、Observable API和簡化的錯誤處理。

HttpClient類別還可以在沒有資料伺服器的情況下使用，這得益於內存Web API封包，該封包模擬了對RESTful API的CRUD操作。儘管HttpClient是開發前端Web應用程式的合乎邏輯的選擇，但實現方式有多種，取決於我們想要充分利用其功能的程度。

在本節中，我們將查看替代方案，並了解如何使用一種更多功能的方法重構我們的應用程式，以替換當前的HttpClient實現。這種方法基於一個專用的HTTP資料服務。

XMLHttpRequest 與 Fetch（與 HttpClient）

Angular的HttpClient類別基於XMLHttpRequest（XHR），這是一個由瀏覽器提供的物件，用於在Web瀏覽器和Web伺服器之間異步傳輸資料，無需重新加載整個頁面。這種技術在2017年之前是唯一的選擇，然後才推出了Fetch API作為其現代替代品。Fetch API旨在提供比XMLHttpRequest API更強大靈活的功能集。下一節將快速查看這兩者，並討論它們的優缺點。

XMLHttpRequest

在1999年，Microsoft 首次推出 Outlook Web Access（OWA）的版本時，引入了 XMLHTTP 概念。XMLHTTP 的發展源於 Exchange 2000 團隊，由 Alex Hopmann 的團隊於1999年實現 IXMLHTTPRequest 介面，隨後由 Internet Explorer 5.0（於1999年3月發布）推出。

Mozilla項目也開發了類似的介面 nsIXMLHttpRequest，並在其 Gecko 佈局引擎中實現。它與 Microsoft 介面相似，但也提供了一個允許透過 JavaScript 使用的包裝器，這就是 XMLHttpRequest，於2000年12月6日在Gecko v0.6上可用。

XMLHttpRequest很快成為主要瀏覽器的事實標準，被實現在Safari、Opera、iCab和Internet Explorer等瀏覽器中。Google的Gmail（2004年）和Google Maps（2005年）是最早採用XMLHttpRequest API的Web應用程序之一，標誌著Web開發進入新時代。

術語”AJAX”（Asynchronous JavaScript + XML）於2005年2月18日首次亮相，由Jesse James Garrett提出。W3C在2006年4月5日發布了XMLHttpRequest對象的第一個草案，試圖成為官方Web標準。

AJAX開發的初期實現在各種瀏覽器間存在差異，但跨瀏覽器JavaScript庫（如jQuery、Axios和MooTools）的出現使開發變得更容易。XMLHttpRequest對象的數據格式由XML轉為JSON、HTML和純文本。Angular的HttpClient類建立在這些發展基礎上，提供了一個方便的包裝器，通過Observable模式實現對XMLHttpRequest的有效使用。

Fetch

在早期，使用原生的XMLHttpRequest物件對大多數網頁開發者而言相當困難，容易導致大量難以閱讀和理解的JavaScript原始碼。這些問題後來被一些程式庫（如jQuery）提供的高層結構解決，雖然解決了問題，但卻帶來了一些不可避免的程式碼和資源開銷。

為了更清晰地解決這些問題，引入了Fetch API，採用內建的基於Promise的方法，使得執行相同的異步伺服器請求變得更加輕鬆，而無需依賴第三方程式庫。

HttpClient

JavaScript-native fetch API和 Angular-native HttpClient 類別都是完全可行的，並且它們都可以在 Angular App 中有效地使用。

以下是使用Fetch的優勢：

  • Fetch是最新的行業標準，用於處理HTTP請求和響應。
  • Fetch是JavaScript原生的，可在Angular以外的任何JavaScript前端框架上使用。
  • Fetch簡化了與Service Workers的工作，因為請求和響應對象與正常代碼中使用的對象相同。
  • Fetch建立在HTTP請求具有單一返回值的基本原則上，因此返回Promise而不是像Observer那樣的流式類型（這在大多數情境下是優勢，但有時也可能成為不利之處）。

以下是使用HttpClient的最相關優勢：

  • HttpClient是Angular原生的，因此得到了框架的廣泛支持和不斷更新。
  • HttpClient允許輕鬆混合和匹配多個Observables。
  • HttpClient的抽象層次使我們能夠輕鬆實現一些HTTP技巧（例如在請求失敗的情況下定義自動重試嘗試）。
  • Observers在某些複雜情境中可能更加靈活和功能豐富，例如執行序列呼叫，以及在發送後能夠取消HTTP請求。
  • HttpClient可以被注入，因此可用於為各種情境編寫單元測試。
  • HttpClient允許使用HttpInterceptors透明地處理HTTP標頭、Bearer令牌和更多基於HTTP的任務。

基於這些原因，我們將繼續使用Angular中的HttpClient。在大多數情境下Fetch API也是一個可行的選擇，看哪一種對於特定任務更適合。

建構資料服務

目前在應用程式中，HTTP調用是放置在組件內的，這對於小規模的應用程式可能是可以接受的，但在實際情況下並非最佳方法。如果我們需要以更複雜的方式處理HTTP錯誤，或者我們需要對取得的資料進行緩存和/或後處理。

我們將會把資料訪問層與資料展示邏輯分開，這可以通過將前者封裝在一個獨立的服務中，然後將該服務注入到組件本身中實現。這正是我們即將要做的。

TypeScript存取修飾符

TypeScript的存取修飾符允許開發人員聲明方法和屬性的可訪問性，包括public、private、protected和read-only。未指定修飾符時，預設為public，可在內外部訪問。private僅在類別內部可訪問，protected在類別及其子類別內部可訪問，read-only在分配後禁止更改屬性值。需注意，這些修飾符僅在編譯時強制執行，無法在運行時阻止不當使用。

新增常用介面方法

  • getData: 這將替換在 CitiesComponent和CountriesComponent 檔案中取得City和Country列表的方法。我們指定了一個新的強類型介面ApiResult<T>。
  • get: 這將替換CityEditComponent和CountryEditComponent檔案中loadData()方法。
  • put和post: 這些方法將替換CityEditComponent和CountryEditComponent檔案中 submit() 方法。
  • getUrl: 這個方法將執行建立API端點的URL，避免在每個 HTTP 請求中使用 environment.baseUrl 屬性。

類型變數和泛型類型 - 和

在get、put和post方法中，我們使用了類型變數<T>而非強類型介面，因為這些方法的返回類型取決於實現它們的衍生類別。選擇使用<T>而非<any>的原因是為了保留函數返回時有關該類型的訊息。<T> 泛型類型允許在聲明和實例化類別或方法時推遲指定返回變數類型，這意味著在實現衍生類別中的方法時能夠捕獲給定參數的類型。這些概念不是新的，因為在後端代碼中已經使用過，而現在由於TypeScript的支持，我們還可以在Angular前端中應用這些概念。

為什麼回傳的是 Observables 而不是 JSON？

在選擇返回 Observable 類型而不是 JSON-based 接口時，我們提到介面類型的限制，相比之下，Observable 集合更加功能豐富。我們無需限制自己及調用這些方法的組件，即使需要執行 HTTP 請求並檢索數據，也可以重新創建 Observable。
