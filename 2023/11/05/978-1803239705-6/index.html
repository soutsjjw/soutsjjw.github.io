探索Angular表單

在上一章中，使用DevTools觀察生成的HTML代碼，可以看到並沒有任何<form>，因為我們並不是使用FormData來送出任何資料，我們使用Angular中的 HttpClient模塊來取得資料，也就是使用異步XMLHttpRequest(XHR)，就是AJAX，這種方法不需要<form>，並且能夠支援以下表單支援的格式處理資料編碼和傳輸任務：

  • application/x-www-form-urlencoded

    表單預設使用的格式，會將內容轉成類似 URL 的 key=value&…

  • multipart/form-data

    表單支援的格式，可以將不同格式的內容包在一起，可以夾帶檔案

  • text/plain

    純文字內容，實際應用中不常出現

範本驅動表單(Template-driven Form)

範本驅動表單將大部分邏輯放在模板代碼中；使用範本驅動的表單意味着：

  • 在 .html 模板文件中構建表單
  • 使用 ngModel 實例將數據綁定到各種輸入字段
  • 使用專用的 ngForm 對象與整個表單相關聯，包含所有輸入字段，每個字段可以通過它們的名稱來訪問

響應式表單(Model-Driven/Reactive Form)

響應式表單是專門為解決範本驅動表單的已知限制而在Angular 2+中新增的。使用這種替代方法實現的表單稱為模型驅動表單或響應式表單，它們實際上是相同的東西。主要的區別在於模板中(幾乎)不發生任何事情，模板只是對一個更複雜的TypeScript對象的引用，該對象在組件類中以編程方式定義、實例化和配置：表單模型。以下為使用模型驅動/響應式的方式重新編寫先前的表單：

  • form 屬性是一個 FormGroup 實例，代表整個表單
  • FormGroup 是一個容器，包含具有相同目的的表單控制項。我們的表單本身就是一個 FormGroup
  • 表單模板中的每個數據輸入元素都由 FormControl 實例表示
  • 每個 FormGroup 實例管理子控制項的狀態，只有當所有子控制項都有效時，它本身才被視為有效

了解資料驗證

將資料驗證添加到表單中是必須的，它用於檢查使用者輸入的準確性和完整性，以提高資料質量並提供使用者良好的體驗。這有助於使用者理解表單無法正常工作的原因以及如何解決問題。

模型驅動驗證

在處理反應式表單時，驗證方法與傳統表單有所不同。總結來說，大部分驗證工作需要在組件類中進行。與在模板中使用HTML屬性添加驗證器不同，我們需要直接在組件類的表單控制模型中添加驗證函數，以便Angular可以在控制的值更改時調用它們。

這種方式允許我們添加同步或異步驗證器。同步驗證器會立即返回一組驗證錯誤或null，而異步驗證器會返回一個已配置的Promise或Observable，以發出一組驗證錯誤或null。這些驗證器函數通常在實例化FormControl時設置。

第一個驗證

我們添加了以下內容：

  • 從 @angular/forms 套件中導入 Validators 類別的參考。
  • 在每個我們的 FormControl 元素上添加了 Validators.required。這個驗證器期望這些欄位有非空值；否則，它將返回無效的狀態。

開啟 city-edit.component.html，在每個mat-form-field元素的結尾加上mat-error元素。

安全導航運算子( ? )

問號是 TypeScript 的安全導航運算子，也稱為 Elvis 運算子，用於防止在屬性路徑中出現 null 或 undefined 值。當使用安全導航運算子時，如果遇到第一個 null 值，TypeScript 將停止評估表達式。這有助於避免在操作屬性時產生空引用錯誤。避免以下一種空引用錯誤之一：

TypeError: Cannot read property ‘required’ of null.
Error TS2531: Object is possibly ‘null’

安全導航運算子允許我們在不確定路徑是否存在的情況下，遍歷對象路徑，並返回路徑的值（如果存在），否則返回 null。這對於需要檢查可為空對象的值非常方便，例如 FormControl 的錯誤、GET 或 POST 參數等。因此，安全導航運算子在這些情況下非常有用。

伺服器端驗證

伺服器端驗證是在資料傳送到後端之後檢查錯誤的過程，與客戶端驗證不同，它在前端檢查資料之前。客戶端驗證速度快，但伺服器端驗證對於確保輸入資料品質是必要的，因為它可以防止許多潛在的問題，如實施錯誤、客戶端入侵和請求偽造等。總之，客戶端驗證是可選的，伺服器端驗證對於關心輸入資料品質的網絡應用是必需的。

可以看到名為Paris的城市共有5個，唯一可以區分的是座標不同。

有些驗證需要檢查新增的城市是否與資料庫中已存在的城市具有相同的名稱、緯經度。這可以避免使用者多次新增相同的城市，但這種驗證無法完全在客戶端實現。為了實現這一功能，需要在伺服器端建立一個自訂驗證器，以非同步方式與伺服器驗證這些值，然後返回有效或無效的結果。這是一個需要在伺服器端執行的驗證任務。

DupeCityValidator

對先前的程式碼所做的重要更改：

  • 添加了一些導入參考，用於實現新的異步自定義驗證器。
  • 建立了名為 isDupeCity() 的方法，其中包含異步自定義驗證器的實現。
  • 配置了新的驗證器用於主要的 FormGroup，即整個表單。

關於自定義驗證器，以下是其功能的摘要：

  • isDupeCity() 方法返回一個 AsyncValidatorFn，它返回一個 Observable。這表示它不返回一個值，而是一個訂閱函數實例，最終會返回一個值，可能是一個鍵/值對象或 null。此值僅在執行 Observable 時發出。
  • 內部函數創建一個臨時城市對象，填充實時表單數據，然後調用一個尚未知道的 IsDupeCity 後端 URL，最終根據結果返回 true 或 null。此處未訂閱 HttpClient，而是使用 pipe 和 map ReactJS（RxJS）操作符進行操作。

由於我們的自定義驗證器依賴於發送到我們的.NET Core後端的HTTP請求，因此我們也需要實現該方法。

我們添加的警報僅在滿足以下三個條件時才顯示：

  • 表單無效
  • 存在與表單本身嚴格相關的錯誤
  • isDupeCity回傳true

我們試著使用前面使用過的資料新增一City：

  • Name: New Tokyo
  • Latitude: 35.685
  • Longitude: 139.7514
  • Country: Japan

自定義驗證器何時會被檢查以及對性能的影響：

  • 異步驗證器只會在所有同步驗證器返回true時才會被檢查。
  • 這個自定義驗證器（isDupeCity）是異步的，它只有在所有FormControl元素中的Validators.required都返回true時才會被調用。
  • 預期isDupeCity驗證器在每次表單提交時可能被調用一次或兩次，這對性能影響是可以接受的。

改善篩選方法

過濾器雖表現的不錯，但每當修改內容後（每次鍵入），都會發送請求，這容易造成巨大的效能問題。

節流和防彈跳

為避免效能問題的產生，考慮以下方案：

  • 即時回應所有通知，這對使用者友好，但會產生效能問題。
  • 立即採取行動，每隔五分鐘檢查訊息。
  • 不立即採取行動，只在過去五分鐘沒有新通知時檢查訊息。

定義

Throttling（節流）是一種行為，限制一個函數在一段時間內能夠被調用的最大次數，以固定的時間間隔執行該函數。

Debouncing（去抖動）是一種技術，防止一個函數在一定時間內不斷被調用，只有在指定的時間內沒有被調用時才執行該函數。

Throttling和Debouncing之間的主要區別在於執行函數的頻率，Throttling以固定的間隔執行，而Debouncing只有在冷卻期過後執行。

為什麼我們要對程式碼進行 Throttling 和 Debouncing？

Throttling和Debouncing主要用於信息技術中的優化和性能優化，特別是在JavaScript中，以有效處理DOM相關的資源密集型任務，如滾動、調整HTML元件和從服務器檢索數據。

在特定情境中，這兩種技術可以用來優化事件處理，以減輕伺服器（控制器和數據庫）的工作負擔，特別是減少不必要的HTTP請求。

在特定用戶體驗情境下，例如過濾器功能，可以使用Debouncing，以推遲HTTP請求直到用戶停止輸入，以提供更好的用戶體驗，同時減少不必要的HTTP請求。

Debouncing對後端的調用

在Angular中實現Debouncing的簡單方法是使用RxJS庫，它允許使用Observables，並使用debounceTime運算符來實現延遲執行操作。

debounceTime運算符在源Observables發射後等待特定時間間隔，然後才發出值，這有助於避免過多的HTTP請求。

使用distinctUntilChanged運算符可以確保只在新值與上一次插入的值不同的情況下發出值，防止重複的HTTP請求。

更新 CitiesComponent

在程式碼中，沒有改變loadData方法，以確保不影響目前的功能。相反，新增了一個名為onFilterTextChanged的方法，用於處理去Debouncing。

onFilterTextChanged方法使用了名為filterTextChanged的變數，這是一個Subject，它是一種特殊的Observable，可以將值傳播給多個觀察者。

簡單來說，每當onFilterTextChanged方法被過濾器的輸入調用時，它會執行以下操作：

  1. 檢查filterTextChanged Subject，確定是否有觀察者正在監聽它。
  2. 如果沒有觀察者，則使用debounceTime和distinctUntilChanged運算符設置去抖動效應，然後訂閱loadData方法。
  3. 向Subject提供新值，這個值將被多播到所有已註冊監聽它的觀察者。
