資料建模方法

Code-First

這是Entity Framework Core自版本4 以來推薦的方法。這種方法的吸引之處在於其前提很容易理解，Code-First方法允許開發人員僅使用標準類別來定義模型物件，而無需使用任何設計工具、XML映射檔案或繁瑣的大量自動生成的程式碼。

採用Code-First的方式意味著在我們的專案內編寫將要使用的資料模型實體類別，並讓Entity Framework相應地生成資料庫。

  • 優點

    • 不需要任何圖表和視覺工具，適用於中小型專案，能節省時間。
    • 具有流暢的程式碼 API，採用慣例優於配置的方式處理常見情境，同時允許開發人員切換到基於屬性的自訂實現，無需自定義資料庫映射。

  • 缺點

    • 需要熟悉C#以及最新的EF慣例。
    • 維護資料庫通常可能會棘手，處理更新而不遺失資料也是如此。從4.3版本開始添加的遷移支援，不斷進行更新以解決這個問題，大大緩解了問題，但同時也對學習曲線產生了負面影響。

Database-First

如果我們已經有現有的資料庫，或者不介意事先建立資料庫，我們可以考慮另一種相反的方法：不是讓EF Core根據模型物件自動建立資料庫，而是使用dotnet ef命令列工具從現有資料庫生成這些物件。這種程式碼生成技術稱為模型脊骨，並依賴於以下命令：

總結來說，選擇「資料庫優先」意味著建立資料庫，然後讓Entity Framework相應地創建/更新其餘部分

  • 優點

    • 如果我們已經有現有的資料庫，這可能是一種值得考慮的方法，因為這將免去重新創建資料庫的需求。
    • 資料遺失的風險將被最小化，因為任何結構變更或資料庫模型更新都將始終在資料庫本身上執行。

  • 缺點

    • 如果我們需要處理叢集、多個實例或多個開發/測試/生產環境，則手動更新資料庫可能會較為棘手，因為我們必須手動保持它們同步，而不是依賴於程式碼驅動的更新/遷移或自動生成的SQL指令。
    • 我們對於自動生成的模型類別的控制較少，因此管理關聯、外鍵和約束將會更加困難。

City

  • [Required]：將屬性定義為必填（非可為空）欄位。
  • [Key]：該屬性承載了資料庫表的主鍵。
  • [Column(TypeName=”decimal(7,4)”)]：該屬性將需要指定類型和精度的資料庫欄位。如果我們不提供此資訊，Entity Framework 將不知道要為這些屬性創建哪些資料庫表欄位的精度，並將回退到其預設值。如果我們的實際資料具有更多的小數位數，這種回退可能導致精度損失。

Entity Framework Core加載模式

  • 即時加載（Eager loading）：相關資料在初始查詢時從資料庫中加載。
  • 顯式加載（Explicit loading）：相關資料在稍後的時間點從資料庫中明確加載。
  • 懶惰加載（Lazy loading）：相關資料在首次訪問實體導航屬性時，會透明地從資料庫中加載。這是三種模式中最複雜的，如果實現不當可能會導致嚴重的性能影響。

實體類型配置方法

  • Data Annotations

    資料註釋是專門用來設定實體類別和屬性的特殊屬性，可以改變預設規則或定義新規則，有助於更好地管理數據定義。

    我們已經在現有的實體類別中使用了資料註釋，加入了 [Key]、[Required] 和 [ForeignKey] 屬性。如果想換配置，需要稍微修改程式碼。

    資料註釋適用於簡單配置的變更，適合小型專案。但不支援 EF Core 的全部配置選項。需要更細緻的實體類型設定時，可能會考慮更強大的方法。

  • Fluent API

    要使用Fluent API，需要在派生的上下文中覆寫 OnModelCreating 方法，使用 ModelBuilder API 來配置模型。

    了解如何使用流暢API的好方法可能是將現有的資料註釋轉換為 ModelBuilder 設定。例如：

    對於我們使用的每個資料註釋，都有對應的Fluent API方法：[Table] 對應 ToTable()，[Key] 對應 HasKey()，[Required] 對應 IsRequired() 等。

    Fluent API的主要好處是它允許我們在不修改實體類別的情況下指定實體配置；此外，Fluent API的配置優先級最高，將覆蓋現有的EF Core慣例和資料註釋。

  • EntityTypeConfiguration classes

    EntityTypeConfiguration類是一個關鍵元素，用於進階編碼模式，克服Fluent API主要問題，保留所有優點。

    簡而言之，這個技術利用ApplyConfigurationsFromAssembly方法，允許在單獨的配置檔案中定義外部規則，減少需要的程式碼。

    了解這方法最好的方式是解釋如何將現有的資料註釋規則轉換為配置類別。

確保連線字串安全

秘密儲存體(Secrets Storage)可以安全有效地存儲任何機密(例如資料庫密碼、API金鑰等)。此功能在開發機的用戶文件夾(Window環境：%APPDATA%\Microsoft\UserSecrets<user_secrets_id>\secrets.json，Linux環境：~/.microsoft/usersecrets//secrets.jso)中創建一個secrets.json文件，可用於以與appsettings.json文件相同的語法添加或覆蓋元素。

使用secrets.json有以下優點

  • 不會被遠程或本地用戶訪問，因為它會在本機文件夾中創建。
  • 會在開發者的個人文件夾中創建，其他本地用戶無法訪問。
  • 能立即使用，擴展 appsettings.json 文件，無需在其中寫入機密信息。

CitiesController

提供(和接收)城市的資料。

  • -name CitiesController：控制器的名稱將為CitiesController
  • -async：為生成的操作啟用非同步
  • -api：生成 API 控制器。
  • -m City：指定模型類別名稱為City
  • -dc ApplicationDbContext：指定 DbContext 類別名稱為ApplicationDbContext
  • -outDir Controllers：生成的控制器檔案將儲存在Controllers目錄中
  • -namespace WorldCitiesAPI.Controllers：指定生成的控制器的命名空間為WorldCitiesAPI.Controllers
